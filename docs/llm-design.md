# LLM 最小实现设计

## 1. 目标

在 1 周内完成最小可交付版本：

- 后端基于现有风险计算结果接入 LLM
- 生成简洁的风险解释文本
- 将解释结果加入现有输出链路
- 前端完成渲染

本阶段目标为：

结构化风险结果 → LLM 解释 → 前端展示

---

## 2. 范围

### 2.1 本阶段包含

- 基于现有风险结果构造 LLM 输入
- 调用单一 LLM 接口
- 生成简洁预警解释
- 将结果接入现有 DTO / WebSocket / API 输出
- 前端渲染解释内容

### 2.2 本阶段不包含

- 多智能体编排
- 复杂 RAG
- 法规知识库检索
- 自动反思与重试
- 论文完整复现

---

## 3. 设计原则

- 风险计算仍由现有规则与算法模块完成
- LLM 仅负责解释与文本生成
- LLM 模块为增强能力，不作为主流程硬依赖
- 实现以最小闭环为优先，避免过度设计

---

## 4. 最小流程

风险计算结果  
→ LLM 解释生成  
→ 输出结果扩展  
→ 前端渲染  

---

## 5. 输入与输出

本阶段仅约束接口方向，不在设计文档中固化具体字段细节。

### 输入

输入来自现有风险计算结果，应包含支持解释生成的关键风险信息。

### 输出

输出为面向前端展示的简洁解释内容，可为：

- 单段文本说明
- 或少量结构化文本块

---

## 6. 集成方式

LLM 模块接入现有风险结果产出链路，位置位于：

风险计算之后，前端消费之前。

调用策略采用最小化原则，仅在存在风险对象时触发解释生成。

---

## 7. 异常处理

LLM 调用失败、超时或返回异常时，不影响主流程运行。

系统应支持以下降级方式之一：

- 返回空解释
- 返回默认模板解释

---

## 8. 实现顺序

1. 打通一次最小 LLM 调用  
2. 接入现有风险输出链路  
3. 前端完成解释展示  
4. 补充异常处理与降级
