> **版本**：v0.5

> **截止日期**：2026-03-16（汇报日）
> **作者**：xin

## 一、v0.5 目标定义

### 做什么

1. **打通 AIS 数据通路**：MQTT 广播 → 后端 → 前端 2.5D 海图实时渲染
2. **实现静态规则危险预警**：基于 if-else 规则，计算本船安全范围和预警信息，推送前端
3. **本船航线展示**：从数据库获取预规划航线，推送前端渲染

### 不做什么（边界）

- ❌ 不做 AI 模型训练和推理（延后到 v1.5）
- ❌ 不做目标船路径预测（延后到 v1）
- ❌ 不做前端重构（直接使用现有 2.5D 海图）
- ❌ 不做 Kafka 削峰（延后到 v2，但 WebSocket 在 v0.5 直接引入）
- ❌ 不做 Redis 缓存（延后到 v2）

## 四、核心模块设计

### 模块 1：AIS 数据订阅与转发

**职责**：订阅 MQTT 广播，解析 AIS 数据，通过 WebSocket 实时推送给前端

```
MQTT Broker ──(subscribe)──→ map-service ──(WebSocket)──→ Frontend

数据流：
  1. map-service 启动时订阅 MQTT Topic（本船 + 目标船 AIS）
  2. 收到 AIS 消息 → 解析为 Ship 对象
  3. 区分本船 / 目标船
  4. 通过 WebSocket 推送给前端
```

**关键数据模型（Draft）：**

```java
public class AisMessage {
    private String mmsi;          // 船舶唯一标识
    private Double longitude;     // 经度
    private Double latitude;      // 纬度
    private Double speed;         // 航速（节）
    private Double course;        // 航向（度）
    private Double heading;       // 船首向（度）
    private Long timestamp;       // 时间戳
}
```

**WebSocket 推送格式（Draft）：**

```json
{
  "type": "AIS_UPDATE",
  "data": {
    "ownShip": { "mmsi": "...", "lon": 114.3, "lat": 30.5, "speed": 12.5, ... },
    "targetShips": [
      { "mmsi": "...", "lon": 114.5, "lat": 30.6, "speed": 8.2, ... },
      ...
    ]
  }
}
```

### 模块 2：静态规则危险预警

**职责**：基于本船和目标船的 AIS 数据，实时计算预警信息

**v0.5 规则（if-else，不涉及 AI）（Draft）：**

```
规则 1：最近会遇距离（CPA）预警
  → 计算本船与每艘目标船的 CPA（Closest Point of Approach）
  → CPA < 阈值（如 0.5 海里）→ 触发预警

规则 2：最近会遇时间（TCPA）预警
  → TCPA < 阈值（如 10 分钟）且 TCPA > 0（正在接近）→ 触发预警

规则 3：安全范围
  → 基于本船航速和航向，计算安全领域（Ship Domain）
  → 目标船进入安全领域 → 触发预警

预警等级：
  GREEN  → 安全
  YELLOW → 需关注（CPA < 1.0 nm）
  RED    → 危险（CPA < 0.5 nm 且 TCPA < 10 min）
```

**预警推送格式（Draft）：**

```json
{
  "type": "WARNING_UPDATE",
  "data": {
    "warnings": [
      {
        "targetMmsi": "413xxxxxx",
        "level": "RED",
        "cpa": 0.3,
        "tcpa": 480,
        "message": "目标船 413xxx 将在 8 分钟后于 0.3 海里处交会"
      }
    ],
    "ownShipDomain": {
      "center": [114.3, 30.5],
      "semiMajor": 0.5,
      "semiMinor": 0.3,
      "orientation": 45.0
    }
  }
}
```

### 模块 3：航线展示

**职责**：从数据库获取本船预规划航线 + 调用路径规划模块接口

```
数据流（v0.5 简化版）：
  1. 前端连接 WebSocket 后，请求本船航线
  2. map-service 查询数据库中预规划航线
  3. （可选）调用路径规划模块接口获取实时航线
  4. 通过 WebSocket 推送给前端

航线推送格式（Draft）：
{
  "type": "ROUTE_UPDATE",
  "data": {
    "ownRoute": {
      "waypoints": [
        { "lon": 114.3, "lat": 30.5, "eta": "2026-03-16T10:00:00Z" },
        { "lon": 114.8, "lat": 30.8, "eta": "2026-03-16T12:00:00Z" },
        ...
      ]
    }
  }
}
```

## 五、技术选型

| 组件        | 选型                                   | 理由                    |
|-------------|---------------------------------------|-------------------------|
| 后端框架    | Spring Boot 3.x                        | 已有基础，生态成熟       |
| MQTT 客户端 | Eclipse Paho / Spring Integration MQTT | Listener 已在用         |
| WebSocket  | Spring WebSocket（内置）                | 实时推送，替代 HTTP 轮询 |
| 数据库      | PostgreSQL + PostGIS                   | 已有，GIS 空间查询能力   |
| ORM        | MyBatis / Spring Data JPA              | 选一个，推荐 MyBatis     |
| 构建工具    | Maven                                  | 已有                    |

### 为什么 v0.5 就引入 WebSocket

```
原因：
  → AIS 是高频实时数据（每秒多条）
  → HTTP 轮询会产生大量无效请求
  → WebSocket 是正确的技术选型，不是优化，是必需品
  → 如果 v0.5 用轮询，v1 再改 WebSocket = 大量重构
  → 不如一步到位
  → 虽然针对单向高频数据流场景，采用基于 HTTP 长连接的 SSE 是更标准的做法。但为了遵循 v0.5 最小化交付且不重构前端的原则，决定沿用既有的 WebSocket 协议。此架构妥协将作为技术债记录在案，留待 v1 前端整体重构时进行降级清理。
```
