# WebSocket LLM 入站与统一 BackendMessage 封装方案

## Summary
- 本次新增“前端通过 websocket 发起 LLM 请求，后端调用现有 `LlmClient`，再把结果封装成 `BackendMessage` 推回前端”的完整链路。
- `PONG`、`LLM` 回复、`LLM` 错误、`RISK_UPDATE` 全部统一复用 `BackendMessage` 外层信封；`RiskObjectDto` 继续作为 `RISK_UPDATE` 的 `payload`，不再单独走手写 `Map` envelope。
- 聊天能力本次按“单轮无状态”实现，但保留 `sequence_id`、`message_id`、`reply_to_message_id`，为后续升级多轮上下文做协议预留。
- 聊天回复和聊天错误只回发给当前 websocket session；风险广播仍然发给所有 session。

## Key Changes
- 在 `dto.websocket` 保持纯协议对象职责：
  - 继续使用 `FrontendMessage { type, message }` 作为前端入站 envelope。
  - 继续使用 `BackendMessage { type, sequence_id, payload }` 作为所有后端出站 envelope。
  - `FrontendChatPayload.sequence_id` / `BackendChatReplyPayload.sequence_id` 明确表示“聊天会话/线程标识”，而 `BackendMessage.sequence_id` 明确表示“服务端出站消息序号”；两者语义不同但都保留。
  - 新增一个聊天错误 payload DTO，字段固定为：`sequence_id`、`message_id`、`reply_to_message_id`、`error_code`、`error_message`、`timestamp`。`timestamp` 用 ISO-8601 字符串。
  - 新增统一消息类型常量或枚举，固定覆盖：`PING`、`PONG`、`CHAT`、`CHAT_REPLY`、`CHAT_ERROR`、`RISK_UPDATE`，避免散落魔法字符串。

- 在 `websocket` 包内拆分“传输层”和“业务层”：
  - `StreamWebSocketHandler` 只负责接收文本、反序列化 `FrontendMessage`、按 `type` 路由，不直接拼装业务 payload，不直接调用具体 LLM client。
  - `StreamWebSocketHandler` 新增定向发送能力：除现有广播外，增加 `sendToSession(session, backendMessage)`；聊天回复和 `PONG` 走定向发送，`RISK_UPDATE` 走广播。
  - `WebSocketService` 收敛成统一出站网关，负责生成服务端 `sequence_id`、封装 `BackendMessage`、序列化并发送。对外暴露明确方法：`sendPong(session)`、`sendRiskUpdate(riskObject)`、`sendChatReply(session, payload)`、`sendChatError(session, payload)`。
  - 未识别的前端 `type` 只记录 `warn` 日志，不主动回包；避免额外扩张公开协议。

- 在 `service.llm` 增加前端聊天编排服务，不让 websocket 层直接依赖 LLM provider 细节：
  - 新增一个面向 websocket 聊天的服务，例如 `LlmChatService`，依赖 `LlmClient`、`LlmProperties` 和 `WebSocketService`。
  - 入参为当前 session 和 `FrontendChatPayload`；内部做最小校验：`content` 非空、`role == USER`，`message_id` 必填，`sequence_id` 透传，`input_type` 允许为空。
  - prompt 构造本次只使用当前消息内容，必要时把 `input_type` 作为提示词元数据附加，但不引入任何后端会话历史存储。
  - 成功时发送 `CHAT_REPLY`，payload 里回填 `sequence_id`、新生成的回复 `message_id`、`reply_to_message_id`、`role=ASSISTANT`、`content`、`source=llm.provider`、`timestamp`。
  - 超时或异常时发送 `CHAT_ERROR`，不静默吞掉；`error_code` 统一使用有限集合，例如 `LLM_TIMEOUT`、`LLM_REQUEST_FAILED`、`INVALID_CHAT_REQUEST`。

- `RiskObject` 统一改为复用 `BackendMessage`：
  - `sendAisMessage` 改成构造 `BackendMessage(type=RISK_UPDATE, payload=RiskObjectDto)` 后广播。
  - 这样前端始终只解析一种外层 envelope，PONG / CHAT / RISK_UPDATE 全部一致，后端发送路径也只保留一套。

## Test Plan
- 反序列化测试：
  - `FrontendMessage(type=CHAT)` 能正确映射出 `FrontendChatPayload`。
  - `FrontendMessage(type=PING)` 能被路由到 heartbeat 分支，不要求 `message` 非空。
- WebSocket handler 测试：
  - 收到 `PING` 时，向当前 session 发送 `BackendMessage(type=PONG)`，不再发送裸 `{"type":"PONG"}`。
  - 收到合法 `CHAT` 时，不广播，且只对当前 session 发送一条 `CHAT_REPLY`。
  - 收到非法 `CHAT`（空内容、非 `USER` 角色、缺少 `message_id`）时，返回 `CHAT_ERROR`。
- LLM 编排测试：
  - 模拟 `LlmClient` 成功返回时，`CHAT_REPLY` payload 字段完整，`reply_to_message_id` 与前端消息一致。
  - 模拟超时/异常时，返回 `CHAT_ERROR`，并区分 `LLM_TIMEOUT` / `LLM_REQUEST_FAILED`。
- 出站统一性测试：
  - `sendRiskUpdate` 输出为 `BackendMessage(type=RISK_UPDATE, payload=RiskObjectDto)`。
  - 所有出站消息都带服务端生成的 `BackendMessage.sequence_id`，并能被 `ObjectMapper` 正常序列化。

## Assumptions
- 心跳语义保持为“前端发送 `PING`，后端回复 `PONG`”；你提到的“回应前端的 pong 消息”按现有代码语义解释为“返回 PONG 响应”。
- 本次聊天协议固定使用 `FrontendMessage.type = CHAT`；如果前端已有别的字面量约定，实施时只替换常量值，不改整体结构。
- 本次不实现后端多轮上下文存储；后续若升级，只在 `service.llm` 新增会话历史读取/裁剪，不改 websocket envelope 和基础 DTO 结构。
- `RiskObject` 这次推荐一并收敛到 `BackendMessage`，这是更低耦合、也更利于前端消费的一条线。
