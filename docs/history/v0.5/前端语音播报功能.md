# 前端 LLM 语音播报与后续主动语音交互扩展方案

## Summary
先实现“前端对 LLM 输出自动语音播报”，并从现在开始按“会话化”设计，为后续接入“点击开始说话 -> 浏览器录音 -> 上传后端 -> whisper.cpp 转文字 -> 喂给 LLM -> WebSocket 回前端 -> 前端显示并 TTS 播放”预留接口与状态结构。

当前阶段默认行为：
- 新的高风险 LLM 输出到达后自动播报
- 前端提供静音/启用语音开关
- 前端只播报“新内容”，不重复读同一条
- 数据模型预留 `conversation_id/session_id`
- 语音播报逻辑与 UI 展示解耦，避免后续录音链路改动时牵扯现有面板

## Key Changes
### 1. 前端语音播报链路
新增浏览器 TTS 服务层，建议放在 `src/services/speechService.ts`：
- 封装 `SpeechSynthesisUtterance`
- 提供 `init()`、`unlock()`、`speak(text, options)`、`stop()`、`pause()`、`resume()`
- 初始化中文 voice，优先 `zh-CN`，其次任意 `zh*`
- 统一处理文本清洗、长度裁剪、异常兜底和浏览器兼容降级

新增播报控制 Hook，建议放在 `src/hooks/useLlmSpeechBroadcast.ts`：
- 顶层挂载于 [App.tsx](/wsl.localhost/Ubuntu/home/xin/workspace/map-system/frontend/src/App.tsx)
- 订阅 store 中的最新 LLM 结果
- 发现“新 LLM 文本”时触发 TTS
- 根据风险等级决定是否抢占当前播报：
  - `ALARM` 可中断当前播报
  - `WARNING` 正常排队或在短节流内忽略
- 不依赖 `RiskExplanationPanel` 是否展开或渲染

### 2. Store 状态设计
在 [useRiskStore.ts](/wsl.localhost/Ubuntu/home/xin/workspace/map-system/frontend/src/store/useRiskStore.ts) 增加与播报相关的独立状态，不和 UI 已读状态混用：
- `speechEnabled: boolean`
- `speechUnlocked: boolean`
- `speechSupported: boolean`
- `spokenLlmMessages: Record<string, string>`
- `lastSpokenAt: Record<string, number>`

推荐新增动作：
- `setSpeechEnabled(enabled: boolean)`
- `setSpeechUnlocked(unlocked: boolean)`
- `markLlmSpoken(messageKey: string, text: string)`
- `clearSpokenState(conversationId?: string)`

消息主键设计：
- 当前阶段先用 `conversation_id + target.id + explanation.text` 组合成稳定 key
- 若后端未来提供 message id，直接切换成 message id

这样可以保证：
- 同一条 LLM 输出只播报一次
- 不因为 UI 点击“已读”就阻止后续播报逻辑
- 后续接主动语音交互时，前后端都能围绕同一会话继续扩展

### 3. 当前 LLM 输出的数据约定
当前前端播报对象仍来自 `target.risk_assessment.explanation`，但建议从现在开始在前端内部标准化成统一事件结构：
- `conversation_id?: string`
- `target_id: string`
- `source: string`
- `text: string`
- `risk_level: string`
- `timestamp?: string`

当前后端即使还没返回 `conversation_id`，前端也应预留字段并使用 fallback：
- `conversation_id = risk_object_id` 或 `default`
- 后续一旦后端提供正式会话 id，可无痛替换

LLM 判定规则：
- 只对明确来自 LLM 的 explanation 进入播报链路
- 推荐前端统一用一个工具函数判断 `source`，不要在多个组件里散落判断逻辑
- 当前接受：
  - `llm`
  - 包含 `llm`
  - 包含 `ai`
  - 包含 `model`
  - 包含 `gpt`

### 4. 浏览器 SpeechSynthesis API 调用方式
封装后内部调用规则如下：
- 创建播报对象：
  - `const utterance = new SpeechSynthesisUtterance(text)`
- 设置参数：
  - `utterance.lang = 'zh-CN'`
  - `utterance.rate = 1`
  - `utterance.pitch = 1`
  - `utterance.volume = 1`
- 绑定 voice：
  - 优先匹配 `zh-CN`
  - 其次匹配 `lang.startsWith('zh')`
- 调用：
  - `window.speechSynthesis.speak(utterance)`
- 控制：
  - `cancel()` 终止
  - `pause()` / `resume()` 控制暂停恢复
  - `getVoices()` 获取语音列表
  - `voiceschanged` 事件用于异步刷新 voice

文本预处理规则：
- 去掉多余空白
- 去除不适合朗读的符号噪声
- 长度裁剪到约 100 到 140 字
- 最终播报文案格式统一为：
  - `目标 {id}，风险等级 {中文等级}。{llm文本}`

### 5. 用户手势解锁与静音控制
由于浏览器常要求用户手势后才允许稳定播报，当前阶段必须提供一个明确的前端开关。
推荐做法：
- 在状态面板或右侧说明区域增加“语音播报”开关
- 用户第一次开启时执行 `speechService.unlock()`
- 成功后将 `speechUnlocked = true`
- 若浏览器不支持，开关置灰并展示“当前浏览器不支持语音播报”

默认策略：
- `speechEnabled = false`
- 用户主动开启后才允许自动播报
- 开启后对新 LLM 结果自动播报
- 用户可随时静音，但不清除已播报记录

### 6. 后续主动语音交互的前端扩展位
为后续“开始说话 -> 录音 -> 上传 -> LLM 返回 -> 显示+播报”预留统一会话模型，前端从现在就按以下方向设计：

建议新增统一消息模型：
- `conversation_id`
- `message_id`
- `role: 'user' | 'assistant' | 'system'`
- `type: 'text' | 'speech' | 'llm_explanation'`
- `content`
- `target_id?`
- `timestamp`

当前阶段虽然只消费后端推来的 LLM explanation，但结构上应允许未来接入：
- 用户点击“开始说话”
- 前端录音 3 到 10 秒
- 上传音频到后端 HTTP 接口
- 后端 whisper.cpp 出文本并送入 LLM
- 后端通过 WebSocket 推回 assistant 消息
- 前端把 assistant 消息同时：
  - 加入会话流
  - 写入 explanation/消息展示区
  - 触发 TTS 播报

也就是说，未来应逐步从“target explanation”过渡为“会话消息 + target 关联”的统一模型；当前版本不必重构 UI，但状态设计要兼容这个方向。

### 7. 推荐的前端模块拆分
建议拆成 3 层，后续扩展最稳：
- `speechService`
  - 只负责浏览器 TTS
- `llmEventNormalizer`
  - 把当前 WebSocket explanation 转成统一 LLM 事件
- `useLlmSpeechBroadcast`
  - 只负责判断“什么时候播报”

这样后续接主动语音输入时，只需要把“后端返回的新 assistant 消息”也丢进同一套 normalizer 和播报链路，不需要重写 UI 或 TTS 服务。

## Test Plan
### 当前阶段
- 浏览器支持语音时，开启语音开关后，新到达的 `WARNING/ALARM` LLM 文本自动播报
- 同一条 explanation 多帧重复出现时，只播报一次
- 新 explanation 替换旧 explanation 时，能再次播报
- `speechEnabled=false` 时不播报，但文本照常显示
- `ALARM` 到达时可中断低优先级播报
- 浏览器不支持 `speechSynthesis` 时，功能静默降级，UI 给出不支持提示
- 页面刷新后，未解锁状态下不自动播报，用户重新开启后恢复

### 与现有 UI 协同
- `RiskExplanationPanel` 的视觉和交互不依赖播报是否开启
- 左侧目标列表与右侧说明面板继续按现有逻辑展示
- 语音播报不会因为面板隐藏或 hover 收起而停止工作

### 后续扩展兼容
- 当前 store 中新增的 `conversation_id/session_id` 相关字段不会影响现有 WebSocket 渲染
- 后续新增“用户主动发消息”后，可与当前播报逻辑共用去重和已播报标记
- 未来后端返回正式 `message_id` 后，播报去重 key 可直接切换，不影响 UI

## Assumptions
- 当前阶段只实现前端对 LLM 输出的 TTS 播报，不实现录音上传
- 默认播报源仍是 WebSocket 回来的 target explanation
- 前端需要自动播报，但必须提供静音/启用开关
- 会话模型从现在开始预留 `conversation_id`，但当前可以先用 fallback 值
- 当前只播报高风险 `WARNING/ALARM` 的新 LLM 内容
- 后续“开始说话”功能将通过新增 HTTP 上传接口和同一 WebSocket 会话回流接入，不推翻现有播报设计
