# 前端语音聊天与模式化转录流程改造方案

## 摘要
- 本方案在现有聊天面板内新增语音录音发送能力，录音通过 `MediaRecorder` 采集，并继续使用 `type: "CHAT"` 的 WebSocket 信封发送。
- 语音消息新增 `mode` 字段，设计支持两种模式：
  - `direct`：语音发送后，由后端完成转录并继续调用 LLM，前端接收转录结果与聊天回复。
  - `preview`：语音发送后，由后端仅完成转录并回传前端，用户确认或修订后再以文本消息发送。
- 当前后端仅具备 `direct` 链路能力；前端保留 `preview` 设计与接口，但入口禁用，不实现独立转录确认框。

## 公共接口与类型
- `ChatRequestMessage` 调整为判别联合：
  - `TEXT`：保留 `content`。
  - `SPEECH`：新增必填 `audio_data`、`audio_format`、`mode`，不再携带 `content`。
- `mode` 取值定义为 `'direct' | 'preview'`；当前前端仅发送 `'direct'`。
- 前端补充 `CHAT_TRANSCRIPT` / `ChatTranscriptPayload` 类型，字段采用 `sequence_id`、`message_id`、`reply_to_message_id`、`transcript`、`language`、`timestamp`。
- 新增录音服务接口，负责权限申请、MIME 类型选择、开始/停止录音、Blob 转 Base64；浏览器音频对象不进入 Zustand。
- 聊天消息视图补充“生成控制占位”状态，用于渲染消息下方的中断按钮预留位；本期按钮固定禁用，不新增中断协议发送逻辑。

## 实现变更
- 聊天输入区替换录音占位按钮，结束录音时提供两种动作：
  - `直接发送`：启用，按 `mode: "direct"` 发送语音。
  - `预览转录`：禁用，并标注“后端暂未支持”。
- 录音实现采用 `navigator.mediaDevices.getUserMedia({ audio: true })` + `MediaRecorder`，优先选择 `audio/webm;codecs=opus` / `audio/webm`；`audio_format` 统一发送 `webm`，`audio_data` 发送原始 Base64。
- 前端新增独立语音工作流状态：`录音中 / 转录中 / 发送完成 / 错误`，不与现有文本发送态混用。
- `direct` 模式链路：
  - 停止录音后创建 `input_type: "SPEECH"` 的用户消息并发送；
  - 本地完成 10 MiB 上限校验；
  - 收到 `CHAT_TRANSCRIPT` 后更新原语音消息文本，用于聊天栏内回显最终转录内容；
  - 后续继续接收同一消息对应的 `CHAT_REPLY` 或 `CHAT_ERROR`。
- `preview` 模式不在本期开放；未来由该模式承担“转录后确认再发送”的交互，不再单独设计额外确认框。
- 每条已成功推送的用户消息与助手回复消息均直接进入聊天栏展示。
- 聊天栏内每条处于生成相关流程中的消息下方增加一个中断按钮占位，表现为转圈动画按钮；当前按钮禁用，仅用于预留未来“主动中断本次生成并发送中断消息”的交互位置。
- 现有重试逻辑维持仅对文本消息开放；语音消息失败后提示重新录音。

## 测试与验收
- 权限路径覆盖：首次授权、拒绝授权、无麦克风、浏览器不支持 `MediaRecorder`。
- `direct` 正常链路覆盖：开始录音、停止录音、发送语音、收到 `CHAT_TRANSCRIPT`、收到 `CHAT_REPLY`，且聊天栏正确回显转录文本与回复消息。
- 异常链路覆盖：WebSocket 未连接、Base64 转换失败、音频超限、后端返回 `CHAT_ERROR`、未收到 `CHAT_TRANSCRIPT` 但已收到 `CHAT_REPLY`。
- 聊天栏中断按钮占位在生成相关消息下方可见、为禁用态，并带转圈动画；点击不产生副作用。
- `preview` 入口应在界面中可见但禁用，且不会触发发送。
- 文本聊天、AI 播报、清空会话及现有非语音重试逻辑不得回归。
- 实施前需先清理现存 `src/components/Dashboard/RiskExplanationPanel.tsx` 未使用参数导致的构建错误，确保 `npm run build` 可作为有效验收。

## 假设与默认项
- 后端正式契约需补充 `mode` 字段及 `CHAT_TRANSCRIPT` 事件定义；当前前端按既有后端实现兼容 `direct`。
- 本次不引入本地转录、流式上传、前端音频转码、语音消息原样重发及中断协议发送。
- 未来若后端补齐 `preview` 与中断协议，前端将在本次预留的模式字段与消息控制位基础上扩展。
