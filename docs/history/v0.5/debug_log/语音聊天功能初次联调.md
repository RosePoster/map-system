# 语音聊天功能初次测试 bug 记录

## 问题概述

语音聊天功能初次联调时，`map-service` 调用本地 `whisper-service` 进行语音转写连续失败，错误现象先后表现为：

- `Unexpected end of file from server`
- `400 Bad Request`
- `Invalid request`

问题涉及 `whisper.cpp` Docker 启动方式、宿主机访问地址以及浏览器录音格式兼容性。

## 现象

### 阶段一

- `map-service` 请求 `http://localhost:8081/inference`
- 后端抛出 `ResourceAccessException`
- 错误信息为 `Unexpected end of file from server`

### 阶段二

- 修正基础连接后，请求地址切换为 `http://127.0.0.1:8081/inference`
- 后端抛出 `400 Bad Request`
- 响应体为 `Invalid request`

## 原因分析

### 1. `whisper-service` 实际未按预期参数启动

Docker 镜像 `ghcr.io/ggml-org/whisper.cpp:main` 的默认 `Entrypoint` 为 `["bash","-c"]`。原始 `compose.yaml` 中使用：

```yaml
command: >
  whisper-server --host 0.0.0.0 -m /models/ggml-base.bin
```

实际运行时仅执行了裸 `whisper-server`，`--host` 与 `-m` 参数未生效。直接后果如下：

- 服务未按预期绑定 `0.0.0.0`
- 服务未按预期加载 `/models/ggml-base.bin`
- 日志回退为默认模型 `models/ggml-base.en.bin`

### 2. `localhost` 存在 IPv6 干扰

宿主机侧访问 `localhost:8081` 时，存在优先命中 `::1` 的情况。为避免 IPv6 路径引入额外不确定性，访问地址统一调整为 `127.0.0.1`。

### 3. 浏览器录音格式与 `whisper-server` 默认输入能力不匹配

前端录音使用 `MediaRecorder`，输出格式为 `audio/webm`。`whisper-server` 默认可直接处理 WAV 类音频；对 `webm` 这类容器格式，需显式开启：

```text
--convert
```

且容器内需存在 `ffmpeg`。在未开启 `--convert` 时，`audio/webm` 请求会稳定返回：

```text
400 Invalid request
```

### 4. 后端未显式透传音频格式

`map-service` 初始实现固定以 `audio.bin` 和 `application/octet-stream` 方式上传音频，未使用前端传入的 `audio_format`。该实现不利于服务端识别真实音频类型，也不利于后续格式扩展。

## 修复内容

### 1. 修正 `whisper-service` 启动方式

在 `compose.yaml` 中显式覆盖 `entrypoint`，并将参数拆分为数组形式：

```yaml
entrypoint:
  - whisper-server
command:
  - --host
  - 0.0.0.0
  - -m
  - /models/ggml-base.bin
  - --convert
```

修复后，容器实际启动命令为：

```text
whisper-server --host 0.0.0.0 -m /models/ggml-base.bin --convert
```

### 2. 统一使用 IPv4 地址

调整 `map-service` 配置：

- `whisper.url=http://127.0.0.1:8081`

并同步修改 `WhisperProperties` 默认值为：

- `http://127.0.0.1:8080`

### 3. 补充 `response_format=json`

后端调用 `/inference` 时显式增加：

```text
response_format=json
```

确保返回结构与 `WhisperResponse.text` 对齐。

### 4. 透传真实音频格式

调整 `WhisperClient` 接口及调用链，新增 `audioFormat` 参数，并在 multipart 请求中：

- 根据 `audio_format` 生成对应文件名，如 `audio.webm`
- 设置对应 MIME 类型，如 `audio/webm`

## 验证结果

### 1. 服务启动验证

- `docker inspect whisper-service` 显示 `Entrypoint` 和 `Cmd` 正确
- 容器日志显示模型加载路径为 `/models/ggml-base.bin`
- 容器内访问 `http://127.0.0.1:8080/` 返回 `200 OK`
- 宿主机访问 `http://127.0.0.1:8081/` 返回 `200 OK`

### 2. 接口行为验证

- `GET /inference` 返回 `404 Not Found`
- 该行为符合接口为 `POST multipart/form-data` 的预期

### 3. 音频格式验证

对同一服务进行对照测试，结果如下：

- `WAV` 请求：`200`
- `WEBM` 请求，未开启 `--convert`：`400 Invalid request`
- `WEBM` 请求，开启 `--convert`：`200`

该结果确认本次 `400` 的直接原因是浏览器录音格式 `webm` 未被服务端转码处理。

### 4. 编译验证

执行：

```bash
./mvnw -q -DskipTests compile
```

编译通过。

## 结论

本次语音聊天初测问题由两个阶段性故障构成：

1. `whisper-service` 启动参数未生效，导致服务未按预期启动并回退到默认模型。
2. 服务恢复后，浏览器录音产生的 `audio/webm` 未启用服务端转码能力，导致 `/inference` 返回 `400 Invalid request`。

修复后，`whisper-service` 已能够正常处理浏览器录音格式的 `webm` 音频，语音聊天链路的基础调用条件已具备。
