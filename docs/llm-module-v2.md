# LLM 模块 v2（后续优化方向）设计

## 1. 文档边界

本文档描述相对 v1 的新增能力与演进方案。

v2 的核心目标是：

- 从“文本解释”升级到“结构化建议输出”
- 从“解释展示”升级到“解释 + 可视化联动 + 动画驱动”
- 从“单点增强”升级到“可验证、可版本化的结构化协议”

职责边界统一说明：

- LLM 不是控制器（Controller）
- LLM 不是最终决策执行器（Executor）
- LLM 仅承担风险解释、规则语义化、场景归类、建议生成、结构化展示指令生成
- 最终规则一致性、安全约束与控制执行，仍由确定性规则模块、策略校验层或执行层负责

---

## 2. 架构升级（增量）

在保留 v1 主链路的前提下，新增上层能力：

- 场景识别与规则语义化
- 结构化建议协议输出
- 地图/面板可视化联动

增量架构：

Sensors -> Deterministic Engine -> Structured Context -> LLM Semantic Advisory Layer -> JSON/UI Advisory -> Visualization & Decision Support

说明：

- Deterministic Engine 继续负责 DCPA/TCPA/相对方位/风险值计算
- LLM Semantic Advisory Layer 负责规则解释、场景归类、动作建议和结构化展示指令生成
- LLM 输出由展示层与决策支持层消费，不直接下发控制指令

---

## 3. 新增建议能力

### 3.1 COLREGs 语义建议

- 对遇（head-on）
- 交叉（crossing）
- 追越（overtaking）

输出角色与策略建议：

- give-way（让路船）
- stand-on（直航船）

### 3.2 解释能力升级

- 输出面向船员的中文可读解释
- 同时给出建议动作与风险等级
- 为告警、面板、语音播报提供统一语义来源

补充：COLREGs 一致性与最终控制合法性不由 LLM 单独保证，仍由确定性规则与策略校验链路兜底。

---

## 4. 输出协议升级（核心）

v2 引入 JSON Schema 约束的结构化输出，作为前后端统一契约。

以下为候选协议草案，仅用于说明输出形态，不代表最终冻结字段定义。

```json
{
  "schema_version": "v2-draft",
  "natural_language_advice": "string",
  "risk_level": "low/medium/high",
  "confidence": 0.0,
  "time_semantics": {
    "valid_window_sec": 120,
    "prediction_horizon_sec": 300,
    "animation_duration_sec": 120
  },
  "reasoning_reference": {
    "colregs_rules": ["Rule 14", "Rule 15"],
    "note": "后续接入法规知识与可追溯链路后进一步收敛"
  },
  "animations": [
    {
      "type": "highlight_target",
      "mmsi": 123456789,
      "cpa_nm": 0.5,
      "tcpa_min": 8,
      "color": "#FF0000"
    },
    {
      "type": "proposed_maneuver",
      "path": null,
      "duration_sec": 120,
      "action": "starboard_20deg"
    }
  ]
}
```

约束（需在实现中强制执行）：

- LLM 不负责生成可执行的地理坐标路径
- `proposed_maneuver.path` 必须由确定性模块（如 CvPredictionEngine / 规则引擎）基于当前船位、航向航速与动作参数计算并填充
- LLM 仅输出语义动作（如 `action: starboard_20deg`）及解释信息；若坐标尚未由确定性模块生成，`path` 应保持空值并由后处理补全

协议要求：

- 文本解释与结构化展示建议分离
- 输出可被 Schema Validation 严格校验
- 无效字段与非法值可在消费层快速拦截
- 坐标类字段（如 `path`）必须标记来源并经确定性链路生成，禁止直接信任 LLM 生成的坐标
- 引入 `confidence` 作为置信度参考，用于 UI 展示、回退策略与人机协同决策参考
- 引入 `schema_version` 进行版本化管理，避免单侧升级导致兼容性问题
- 输出建议具有时间语义，支撑建议有效期、预测窗口与动画时长管理

版本联动要求：

- 协议版本需与生成侧 Prompt 模板联动维护
- 协议版本需与后端校验逻辑联动维护
- 协议版本需与前端消费逻辑联动维护

## 5. 可视化能力升级

v2 在地图端新增以下表现：

- 目标船高亮（MMSI + 风险圈）
- 动态拟规避路径（插值动画 + 箭头）
- CPA/TCPA 连线与时间标签
- 风险热区闪烁
- 未来轨迹预测叠加

建议技术栈：

- 地图渲染：Cesium / Leaflet / OpenLayers
- 动画引擎：Tween / GSAP
- 数据校验：zod / ajv

时序稳定性要求：

- 连续帧下，相似场景建议不应频繁抖动
- 大幅建议变化应具备更高触发门槛或提供额外解释
- 时间维度将直接影响展示稳定性、建议有效期与系统一致性

---

## 6. Generative UI 机制

v2 采用“LLM 输出结构化展示建议而非纯文本”的机制：

- 通过 Structured Output 或 Function Calling 返回动作 JSON
- 前端展示执行器按类型分发到高亮、连线、航线动画等模块
- 结合 Pydantic/Schema 进行生成侧与消费侧双重约束

说明：该机制用于展示驱动与决策支持，不等同于控制执行链路。

---

## 7. 工程化与安全性

### 7.1 调用模式

- 优先采用 Function Calling 返回结构化对象
- 统一协议版本号，支持后续字段扩展

输入安全建议：

- 对 AIS 数据流中的自由文本字段（如船名、目的地）进行 Input Sanitization 与转义
- 防御异常字符导致的 JSON 结构破坏与潜在 Prompt Injection 风险

### 7.2 容错策略

- JSON 解析失败或校验失败时，回退到仅文本解释
- 若 LLM 不可用，回退到传统 ARPA 展示链路
- 设置超时与降级机制，确保主链路可用性

### 7.3 验证路径

1. 模拟环境验证（OpenCPN + AIS 模拟数据）
2. 半实物联调（前端动画与后端决策一致性）
3. 实船测试（受控场景）

---

## 8. 上下文与状态管理（Context & State Management）

LLM 本身是无状态组件。为满足时序稳定性，系统侧需维护必要历史上下文，例如：

- 通过滑动窗口传递过去 N 帧关键态势摘要
- 或传递上一帧已生成的结构化建议

建议引入状态变更阈值（Threshold）：

- 仅在状态变化超过阈值时触发显著展示更新
- 避免微小输入波动导致 UI 频繁闪烁

---

## 9. 调用策略与性能成本约束（待收敛）

当前尚未最终确定调用频率、延迟预算与成本约束，后续需结合真实链路表现收敛。

建议原则：

- 不按每帧调用 LLM
- 高频传感器流与低频语义解释解耦
- 优先事件触发而非固定频率轮询
- 仅在中高风险、风险跃迁或场景显著变化时触发
- 短时间内相似输入优先复用最近一次结果
- 明确超时、重试上限与降级路径

---

## 10. 评测指标（待细化）

当前尚未最终确定评测指标与评测方法，需在实现与实验阶段进一步收敛。

建议性评测框架：

- 结构化输出有效性：JSON 解析成功率、Schema 校验通过率
- 建议质量：场景分类正确率、与规则引擎一致率、文本可理解性
- 时序稳定性：连续帧建议变更率、相似输入一致性
- 系统开销：平均时延、P95/P99 延迟、单次调用成本

---

## 11. 未决设计问题（待收敛）

### 11.1 多目标冲突下的统一生成策略

当前尚未最终确定多目标场景下的统一生成策略。多目标冲突会影响：

- 输出协议组织方式
- 动画分发机制
- 展示逻辑
- 建议聚合方式

候选方案：

- 模式 A（推荐起步）：每个 target 单独评估，后端执行 conflict resolution
- 模式 B（进阶）：LLM 输入全局态势，输出统一 maneuver 建议

### 11.2 动作背离与重新评估机制

当系统检测到实际航向/航速/机动轨迹与当前建议发生显著背离时：

- 立即废弃过期 advice / animation
- 触发新一轮态势解释与建议生成

---

## 12. 分阶段演进建议

1. v2.1：先上线结构化协议草案与校验链路
2. v2.2：上线地图动画执行器与可视化联动
3. v2.3：增强 COLREGs 场景判定、建议质量与时序稳定性
4. v2.4：完善评测指标、成本约束与实船闭环

---

## 13. 一句话定义（v2）

LLM 在 v2 中承担“语义建议层（Advisory Layer）+ 结构化展示指令生成层”，以协议化方式服务人机协同决策支持，而非直接控制执行。
